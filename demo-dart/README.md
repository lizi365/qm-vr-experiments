# Quantum VR — Demo Dart（抓取投掷交互测试）

## 这个demo在测什么，为什么是飞镖而不是乒乓球

Demo 0 和 Demo 1 测的都是**GPU渲染负载的上限**（能塞多少粒子、能跑多稳的FPS），这条线已经验证过了。但VR体验的"好不好玩"不只取决于渲染性能，还取决于**交互本身的手感**——抓取的时机是否符合直觉、投掷的轨迹是否符合物理直觉、命中判定是否准确。这条线在这个项目里之前完全没测过，是一个独立于GPU负载的、全新的验证维度。

选飞镖而不是乒乓球（挥拍击打）或者投篮（更复杂的抛物线+篮筐碰撞），是因为**飞镖是"抓取→蓄力→释放"这个动作链条里最简单、最容易分离出问题的场景**：只有一次性的抓、甩、放，没有连续挥拍的时序判断，也没有篮筐那种需要考虑入射角度的复杂碰撞。如果连"抓一个东西、扔出去、砸中一个平面靶"这个最基础的动作都测不准，那更复杂的交互（挥拍、投篮、后续可能设计的"用手势操控量子实验装置"之类的核心玩法）就更没有验证的意义了。这个demo本质上是在给后续所有需要"抓取+释放"类交互的正式内容打地基。

## 按键说明（当前版本，2026-07-10 第三轮之后）

这个demo的按键映射改过两次，容易搞混，测试前先看这张表：

| 操作 | 按键 | 说明 |
|---|---|---|
| 抓取/投掷飞镖 | **扳机（trigger，食指）** | 之前是抓握键(squeeze/grip)，2026-07-10改成trigger，因为更符合"抓取/发射"类动作的生理直觉 |
| 退出VR | **抓握键（squeeze/grip，中指侧键）** | 之前是扳机，跟抓取键对调了——两个功能不能用同一个键，所以退出改成了squeeze |

`GRAB_BUTTON_INDEX = 0` 是代码里假设trigger对应的按钮索引（WebXR "xr-standard"手柄映射的通用约定），**这次改动加了实测用的console.log**（`logGamepadButtonChanges`），进VR后按任意键，控制台会打印是哪个`buttons[i]`在响应，可以用来验证这个假设在Pico上是不是真的成立，不用再凭猜测。

## 场景里有什么

- **飞镖靶**：固定在前方的一块圆形靶面，Canvas贴图画出同心圆环（靶心金色10分，往外依次红/米白/绿/米白，对应8/6/4/2分，脱靶0分），贴在一个Plane上，没有做复杂建模
- **飞镖模型**：闲置时悬浮在场景中一个固定的中性位置（双手都够得着，不预设是哪只手），有轻微上下浮动，暗示"还没抓稳"
- **悬浮记分面板**：靶上方，跟Demo 1同样的视觉方案（深色圆角背景+青色文字），显示本次得分/累计得分/已投掷次数/上次投掷初速度/**上次命中距靶心的原始距离**/FPS/当前状态
- **命中标记点**：飞镖每次穿过靶平面时，会在精确碰撞位置生成一个跟环数同色的小球标记，停留2秒后淡出，最多同时保留3个，方便肉眼核对命中点跟评分是否对得上
- **场景内退出VR面板**：视野右下方外侧一块小面板，手柄射线指向+**抓握键**点击即可退出VR（见下方修复说明）
- **预抓取发光提示**：**扳机**压力超过30%但还没到抓取阈值时，镖身会有一个呼吸感的高亮，告诉你"已经在可抓取范围了，继续按"
- **音效反馈**：抓取瞬间有短促的"咔"声，命中靶面按环数分级触发不同音调（外环闷、内环脆），10环额外叠加一段欢呼音效——全部用Web Audio API振荡器/白噪声合成，没有引入外部音频文件
- **命中评级文字**：6环以上（GOOD/GREAT/EXCELLENT）会在命中点弹出一个短暂的评级文字，带弹出缩放动画，2/4分和脱靶只有标记点、不弹字

## 操作方式（VR模式）

1. 伸手靠近飞镖（不用管是左手还是右手），按住手柄的**扳机键（trigger，食指）**——2026-07-10从抓握键改成了扳机键，因为食指扣动更符合"抓取/发射"类动作的生理直觉。不需要按到底，压力过了一个较低的阈值就会触发抓取（见下方修复说明），按到30%左右镖身会先发光提示你"已经在范围内了"
2. 按住不放，甩动手柄蓄力——大幅度的手臂挥动和小幅度的手腕甩动都可以，两种都能可靠触发投掷
3. 松开扳机键，飞镖会按刚才甩动的速度矢量+重力飞出去
4. 飞镖穿过靶所在的平面时会自动判定命中位置并计分（同时在命中点生成一个标记球），然后自动飞回场景中的中性待抓取位置，可以立刻用任意一只手开始下一次投掷
5. 需要退出VR时，手柄射线指向右下方的"退出VR"面板，按**抓握键（squeeze/grip）**即可——注意这跟抓取飞镖的键不一样，别按混了

**桌面模式**：没有VR手柄，没法模拟"抓取+甩动+释放"这套动作，所以桌面模式下只会静止展示靶和飞镖模型，不能投掷。这是有意为之——用鼠标模拟投掷手感毫无参考价值，不如直接说清楚这个demo只面向VR手柄测试。

## 速度计算逻辑（如果实测手感不对，从这里开始查）

飞镖释放瞬间的初速度由两部分相加：

1. **线性速度**：**最近5帧手柄位置的滑动窗口**（2026-07-10从8帧缩短到5帧，提高响应），取窗口里最早一帧和最新一帧的位置差，除以这两帧之间的时间差，得到"释放前这段时间的平均速度"。没有只用最后一帧的瞬时位移——那样太容易被单帧的追踪抖动带偏方向。
2. **旋转带来的切向速度**：手腕甩动的特征是"朝向变化快、位移距离短"，纯线性位移窗口对这种动作天然不敏感——这是2026-07-10发现的问题（见下方修复说明）。现在同时用同一个窗口算出手柄的角速度（旋转轴+角速度大小），再乘上飞镖抓取点相对手柄的偏移向量（叉乘），算出"因为手柄在转动，飞镖被甩出去的那部分速度"，跟线性速度相加。这是标准的刚体运动学公式：`v_point = v_origin + ω × r`。

面板上会显示"上次初速度"这个数值（单位 m/s），**这是专门留出来给你做手感校准用的**：如果你觉得自己甩得很轻，但数值显示速度很大（或者反过来），说明速度计算逻辑有问题，需要回头调整：
- 滑动窗口的帧数（`HISTORY_LEN`，目前是5帧）——窗口太长会把"蓄力阶段的静止"也算进平均速度里，导致数值偏低；窗口太短又会重新变得对抖动敏感
- 是否需要对计算出来的速度做一次缩放校准（目前完全没有做人为放大/缩小，重力也是标准的9.8 m/s²，都是诚实的物理量，没有为了"看起来好扔"而调参）

## 这次修的三个真机测试反馈问题（2026-07-10）

**1. 抓取手不再预设左右**：之前飞镖硬编码"默认挂在controller0"，在Pico 4 Ultra Enterprise上controller0固定对应左手，导致惯用右手的用户根本抓不到飞镖。现在改成：飞镖闲置时停在场景里一个固定的中性位置（不属于任何一个controller index），哪只手先按抓握键就归哪只手；当前持镖手松开、飞镖飞出去、计分、回到中性位置之后，另一只手才能触发下一次抓取——同一时刻只能有一支飞镖被持有/飞行，不会两只手抢一支镖。

**2. 加了命中诊断可视化**：之前反馈"分数随位置变化能感受到，但计算可靠性不确定"——光看一个环数数字，没法判断是碰撞检测本身算错了，还是自己对空间位置的主观感觉有偏差。现在：
- 飞镖穿过靶平面的精确碰撞点会生成一个跟命中环数同色的小球标记（金色=10分，红色=8分，以此类推，脱靶是灰色），停留2秒后淡出，最多同时保留3个方便对比
- 面板上新增一行"上次命中距靶心"的原始距离数值（米，不做换算美化），可以直接拿这个数字去对照 `RINGS` 数组里的环半径分界（0.08/0.22/0.42/0.65/1.0，都是相对`TARGET_RADIUS`的比例），验证评分区间划分是否合理

**3. 加了场景内退出VR的方式（这是三个反馈里最重要的一个）**：VRButton默认生成的"EXIT VR"是页面上的DOM按钮，一旦进入沉浸模式用户根本看不到浏览器界面，这个按钮实际不可达——之前demo0/demo1/demo-dart三个页面都有这个问题，进去之后没有能用的退出方式。现在做了一个通用组件 `shared/xrExitButton.js`，三个demo都接入了：视野右下方外侧有一块3D面板，手柄射线指向它、按扳机键（trigger，不是抓握键，避免跟demo-dart的抓取手势冲突）就会调用 `renderer.xr.getSession().end()` 优雅退出。面板位置选在"平时看东西不会碰到、但低头瞟一眼右下就能看到"的地方，没做成正对视野中央那种容易误触的位置。

## 这次修的输入响应灵敏度与交互反馈问题（2026-07-10 第二轮）

**1. 抓取响应不灵敏**（反馈"按键后经常没能响应"）：之前只依赖WebXR原生的`squeezestart`/`squeezeend`事件——这两个事件的实际触发时机由浏览器/运行时自己判定，不透明，也不一定够灵敏，等于"什么时候算按下了"这件事完全不在我们控制范围内。现在改成：如果控制器的WebXR Gamepad暴露了模拟量（`button.value`，0~1连续值，而不是只有布尔的`pressed`），每帧轮询，用一个明显低于"按到底"的阈值（`GRAB_THRESHOLD = 0.6`）触发抓取，响应更快也更可控；原生事件继续保留作为兜底（万一某个输入源真的拿不到模拟量）。同时加了"预抓取"反馈：压力超过`PRE_GRAB_THRESHOLD = 0.3`但还没到抓取阈值时，镖身会有一个呼吸感的发光提示，减少"按了但不知道有没有生效"的挫败感。

**2. 手腕小幅度投掷动作捕捉不到**（反馈"应该更好地去捕捉大动作，避免失败过多"——实际现象是小幅快速动作被漏判，不是要弱化大动作识别）：原来的8帧滑动窗口只算线性位移，对"手腕甩动"这种"朝向变化快、位移距离短"的动作不友好——纯线性位移窗口会把这种动作的力道稀释成一个很小的速度值，达不到有效投掷的门槛。现在做了两处调整：滑动窗口从8帧缩短到5帧（提高响应），并且新增了角速度的贡献——用刚体运动学公式`v_point = v_origin + ω × r`把"手柄转动导致飞镖被甩出去"的那部分切向速度也算进去（详见上方"速度计算逻辑"）。这个公式对大幅度手臂投掷（转动慢，这一项贡献很小）和小幅度手腕甩动（转动快，这一项能补足线性位移算出来的"虚弱"速度）是自然兼顾的，不需要写"看起来是大动作还是小动作"这种启发式判断分支。

**3. 交互反馈层太单薄**（之前只有面板数字变化）：加了音效和视觉两层反馈，都用Web Audio API振荡器/白噪声合成，没有引入外部音频文件依赖：
- 抓取成功：短促降调的"咔"声（方波+快速包络）
- 命中靶面：按环数分级触发不同音调，外环闷（低频sine），内环脆（更高频、triangle音色更亮），10环额外叠加一段带通滤波白噪声模拟的欢呼声
- 视觉上，6环以上会在命中点弹出评级文字（GOOD/GREAT/EXCELLENT），Canvas贴图+简单的弹出缩放动画，1.5秒后连同标记点一起淡出；2/4分和脱靶只有命中标记点，不额外弹字，避免每次投掷都弹一堆文字造成视觉噪音

## 这次修的按键映射与反馈层排查（2026-07-10 第三轮）

**1. 抓取键从抓握键(squeeze/grip)改成扳机(trigger)**：第二轮测试已经确认squeeze的响应机制work得很好（左右手都顺畅），但用户反馈trigger更符合"抓取/发射"类动作的生理直觉——食指扣扳机比中指侧键按压更接近直觉里的"抓东西"动作。第二轮调好的响应机制（阈值滞回0.6/0.45、预抓取发光0.3）原样保留，只是把监听的物理按键源从`buttons[1]`(squeeze)换成`buttons[0]`(trigger)，原生事件也从`squeezestart`/`squeezeend`换成了WebXR规范里trigger对应的标准事件名`selectstart`/`selectend`。**这次没有能力上真机测试，`GRAB_BUTTON_INDEX=0`仍然是标准映射的假设值，没有实测验证过**——已经按你的要求加了`logGamepadButtonChanges`日志（见上方"按键说明"），下次测试时留意控制台输出确认这个假设成不成立。

**2. 退出按钮同步换成抓握键，避开冲突**：`shared/xrExitButton.js`之前特意选trigger是为了避开demo-dart的抓取手势（当时抓取用squeeze），现在抓取改成trigger了，两者会撞在一起，所以退出按钮同步改成了squeeze。确认过这个改动对demo0/demo1没有副作用：demo0没有绑定任何自定义手柄事件；demo1的粒子档位切换绑的是`selectstart`(trigger)，跟squeeze完全不冲突。

**3. 排查音效完全没反应——找到了确凿的根因并修复**：AudioContext的`resume()`只有在真正的"用户手势"调用栈里发起才会生效，这是浏览器的自动播放限制。走查代码发现：之前唯一调用`ensureAudio()`的地方是`grabDart()`，而`grabDart()`同时被两条路径调用——原生的`selectstart`事件（真实手势）和每帧轮询的`updateAnalogGrabInput()`（只是渲染循环里的普通函数调用，不算用户手势）。因为轮询每帧都在跑，几乎总是比原生事件先摸到抓取阈值、抢先调用`grabDart()`；`grabDart()`内部有状态守卫，抢到的那次执行完，原生事件触发时状态已经不是IDLE了，直接return——`playGrabSound()`根本没机会在"真手势"的调用栈里跑。`resume()`在非手势上下文里静默失败、不报错，现象就是"完全没有声音、控制台也没有报错"，跟你的猜测完全一致。**已修复**：把第一次`resume()`尝试挪到"Enter VR"按钮的click事件里——这是唯一100%保证是真实用户手势的时机。同时在`ensureAudio()`里加了`console.log`打印`audioContext.state`的变化，下次测试能直接在控制台确认状态是不是真的变成了`running`。

**4. 排查视觉反馈（GOOD/GREAT/EXCELLENT）没出现——代码走查没找到同等确凿的bug**：跟音效不一样，`resolveThrow()`→`spawnRatingText()`→贴图生成→`scene.add(mesh)`这条调用链逐行看下来没发现明显的"接错线"问题。老实说，这次没有设备可以实测，不能排除有一个还没发现的隐藏bug；但也有一个同样成立的可能性——评级文字只有命中6/8/10环才会弹（2/4分和脱靶只有标记点），如果当时的测试投掷因为抓取/速度问题还不够准，根本没打到6环以上，那视觉反馈"没出现"就是设计上的正常行为，不是bug。**没有实机验证不能瞎猜，所以这次没有重写这块逻辑**，而是按你的要求加了日志：`resolveThrow()`里打印`scoreValue`和`hitPoint`，`spawnRatingText()`里打印是否找到了对应的贴图、mesh有没有成功add到scene。下次测试时如果日志显示函数确实被调用了、mesh也生成了，但VR里还是看不见，那就是真的有渲染层面的bug（比如位置计算飘到视野外之类），到时候可以照着日志的实际数值排查；如果日志显示`scoreValue`每次都小于6，那就说明只是测试投掷没打到高分区，不是代码问题。

## 已知的简化 / 局限

- **扳机键在xr-standard手柄映射里假设是按钮索引0**（`GRAB_BUTTON_INDEX = 0`，索引1是抓握键squeeze，现在给退出按钮用）——这是WebXR标准手柄profile的通用约定，2026-07-10这次改动加了`logGamepadButtonChanges`实测日志（见上方"按键说明"），下次测试时留意控制台输出，如果"预抓取发光提示一直不亮"或者索引对不上，日志里能直接看到实际按下的是哪个`buttons[i]`
- 碰撞检测是逐帧检查飞镖是否穿过靶所在的平面，不是连续碰撞检测——如果初速度极快导致单帧内穿越距离过大，理论上可能出现"隧穿"漏判的情况，先不处理，除非实测发现这是个真问题
- 没有接入 cannon.js 之类的完整物理引擎，重力+速度是手写的逐帧积分，够用但不是精确仿真
- 两只手柄不能同时抓住飞镖，飞镖同一时刻只能被一只手持有
