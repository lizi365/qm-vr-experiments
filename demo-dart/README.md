# Quantum VR — Demo Dart（三关抓取投掷小游戏）

> 2026-07-11：从单一靶场测试升级成了红白机(FC/NES)式的三关小游戏——目标清晰、
> 用生命值而不是一击结束来降低失败成本、难度曲线靠时机判断而不是复杂操作。
> 完整设计说明见下方"三关小游戏设计"一节。最早期的"这只是个交互测试demo"
> 定位描述还留着，因为技术验证的初衷（抓取/投掷手感）没有变，只是现在

## 这个demo在测什么，为什么是飞镖而不是乒乓球

Demo 0 和 Demo 1 测的都是**GPU渲染负载的上限**（能塞多少粒子、能跑多稳的FPS），这条线已经验证过了。但VR体验的"好不好玩"不只取决于渲染性能，还取决于**交互本身的手感**——抓取的时机是否符合直觉、投掷的轨迹是否符合物理直觉、命中判定是否准确。这条线在这个项目里之前完全没测过，是一个独立于GPU负载的、全新的验证维度。

选飞镖而不是乒乓球（挥拍击打）或者投篮（更复杂的抛物线+篮筐碰撞），是因为**飞镖是"抓取→蓄力→释放"这个动作链条里最简单、最容易分离出问题的场景**：只有一次性的抓、甩、放，没有连续挥拍的时序判断，也没有篮筐那种需要考虑入射角度的复杂碰撞。如果连"抓一个东西、扔出去、砸中一个平面靶"这个最基础的动作都测不准，那更复杂的交互（挥拍、投篮、后续可能设计的"用手势操控量子实验装置"之类的核心玩法）就更没有验证的意义了。这个demo本质上是在给后续所有需要"抓取+释放"类交互的正式内容打地基。

## 按键说明（当前版本，2026-07-10 第四轮之后）

这个demo的按键映射改过好几次，容易搞混，测试前先看这张表：

| 操作 | 按键 | 说明 |
|---|---|---|
| 抓取/投掷当前物体 | **扳机（trigger，食指）** | 之前是抓握键(squeeze/grip)，第三轮改成trigger，因为更符合"抓取/发射"类动作的生理直觉 |
| 切换投掷物类型 | **抓握键**，射线指向左下方面板 | 第四轮新加，见下方"三种投掷物"说明 |
| 退出VR | **抓握键**，射线指向右下方面板 | 第三轮从扳机改成抓握键，避开抓取手势冲突 |

**切换投掷物和退出VR用的都是抓握键**——这不是失误，是刻意的：trigger已经被"抓取当前物体"占用了（对着任意方向扣扳机都会尝试抓取，没有朝向限制），如果选择面板也用trigger点击，指着菜单选东西的同时会同时触发一次抓取，两个操作会打架。抓握键上挂了两块面板，但左下（选择物体）和右下（退出VR）离得够开，同一根射线不会同时扫到两块，不会互相误触。**这跟最初"复用退出按钮同一套trigger点击模式"的建议不完全一致**，如果后续测试发现两块面板还是容易混淆，可以考虑换成别的区分方式（比如给选择面板加个短暂的确认延迟）。

`GRAB_BUTTON_INDEX = 0` 是代码里假设trigger对应的按钮索引（WebXR "xr-standard"手柄映射的通用约定），加了实测用的console.log（`logGamepadButtonChanges`），进VR后按任意键，控制台会打印是哪个`buttons[i]`在响应，可以用来验证这个假设在Pico上是不是真的成立，不用再凭猜测。

## 场景里有什么

- **昏暗森林场景**：深绿偏黑的背景+浓雾，程序化Canvas噪声贴图做的斑驳落叶/苔藓地面，周围5棵简化树干剪影（圆柱+多层圆锥），营造视线受限的林间氛围，细节见下方"森林氛围"说明
- **飞镖靶**：固定在前方的一块圆形靶面，Canvas贴图画出同心圆环（靶心金色10分，往外依次红/米白/绿/米白，对应8/6/4/2分，脱靶0分），贴在一个Plane上，没有做复杂建模
- **三种可选投掷物**：飞镖/番茄/斧头，左下方面板抓握键切换，细节见下方"三种投掷物"说明
- **飞行尾焰**：投掷物飞行时尾部有火星粒子拖尾，飞得越快尾焰越密集，颜色按投掷物类型区分
- **闲置时悬浮**：当前选中的投掷物闲置时悬浮在场景中一个固定的中性位置（双手都够得着，不预设是哪只手），有轻微上下浮动，暗示"还没抓稳"
- **悬浮记分面板**：靶上方，跟Demo 1同样的视觉方案（深色圆角背景+青色文字），显示本次得分/累计得分/已投掷次数/上次投掷初速度/**上次命中距靶心的原始距离**/FPS/当前状态
- **命中标记点**：投掷物每次穿过靶平面时，会在精确碰撞位置生成一个跟环数同色的小球标记，停留2秒后淡出，最多同时保留3个，方便肉眼核对命中点跟评分是否对得上
- **场景内退出VR面板**：视野右下方外侧一块小面板，手柄射线指向+**抓握键**点击即可退出VR
- **预抓取发光提示**：**扳机**压力超过30%但还没到抓取阈值时，当前投掷物会有一个呼吸感的高亮，告诉你"已经在可抓取范围了，继续按"
- **音效反馈**：抓取瞬间有短促的"咔"声，命中效果按投掷物类型区分（见下方"三种投掷物"说明），全部用Web Audio API振荡器/白噪声合成，没有引入外部音频文件
- **命中评级文字**：6环以上（GOOD/GREAT/EXCELLENT）会在命中点弹出一个短暂的评级文字，带弹出缩放动画，2/4分和脱靶只有标记点、不弹字（三种投掷物通用，评分逻辑完全一致，见下方说明）

## 操作方式（VR模式）

1. 伸手靠近当前选中的投掷物（不用管是左手还是右手），按住手柄的**扳机键（trigger，食指）**——不需要按到底，压力过了一个较低的阈值就会触发抓取，按到30%左右会先发光提示你"已经在范围内了"
2. 按住不放，甩动手柄蓄力——大幅度的手臂挥动和小幅度的手腕甩动都可以，两种都能可靠触发投掷
3. 松开扳机键，投掷物会按刚才甩动的速度矢量+重力飞出去，飞行途中带火星尾焰
4. 投掷物穿过靶所在的平面时会自动判定命中位置并计分（同时触发通用的标记球/评级文字反馈，以及该类型特有的命中效果），然后自动飞回场景中的中性待抓取位置，可以立刻用任意一只手开始下一次投掷
5. 需要切换投掷物类型时，手柄射线指向左下方的选择面板，对准飞镖/番茄/斧头其中一项，按**抓握键**即可切换（持有/飞行中不能切换，必须等当前这一次投掷结束）
6. 需要退出VR时，手柄射线指向右下方的"退出VR"面板，按**抓握键**即可

**桌面模式**：没有VR手柄，没法模拟"抓取+甩动+释放"这套动作，所以桌面模式下只会静止展示靶和当前选中的投掷物，不能投掷、不能切换类型。这是有意为之——用鼠标模拟投掷手感毫无参考价值，不如直接说清楚这个demo只面向VR手柄测试。桌面模式下按**L键**可以调试性地扣一条命（见下方"生命值"），方便在没有头显的情况下也能验证扣命/闪光/重开这条流程。

## 三关小游戏设计（2026-07-11 第五轮）

### 整体架构

- **3条命**：视野左上角常驻显示心形图标（Canvas贝塞尔曲线画的，没用emoji字体，避免不同头显渲染不一致），扣完显示"再来一次"提示，按扳机键重开**当前所在的这一关**（不是退回第一关）。
- **连击计分**：命中判定分>=6分（`COMBO_QUALIFY_SCORE`）时连击数+1，倍率按`COMBO_TABLE = [1.0, 1.2, 1.5, 2.0]`（对应连续命中1/2/3/4次及以上）递增；一旦脱靶或命中<6分，连击清零、倍率回到1.0。这一杆的实际得分 = 原始命中分 × 当时的倍率，四舍五入取整。
- **进度条**：视野正上方，显示"第N关 · 关卡名"+进度条+`已获得分/通关阈值`+当前连击倍率（倍率>1时才显示）。填满立即触发过关，不需要等计时结束。
- **过关反馈**：全屏白色闪光（快速淡入+较慢淡出）+ 5音符8-bit风格上扬琶音（C5-E5-G5-B5-D6方波，每个音符间隔70ms）。
- **失败反馈**：全屏红色闪光 + 低沉下行音效（220Hz降到100Hz的锯齿波）。
- **关卡转场**：过关闪光+音效播完约500ms后，一块不透明深色Plane从左侧扫入盖住整个视野、在完全盖住的那一帧切换关卡内容（场景基调/敌人或靶/投掷物类型全部在这一帧换掉）、再继续扫到右侧离开——玩家看不到"啪"一下的生硬切换瞬间，全程约0.9秒。

### 得分公式

```
这一杆的原始分 rawScore：
  第一关(靶场)：scoreForDistance()算出来的环数分 —— 10/8/6/4/2/0，逻辑完全没动
  第二/三关(敌人)：命中敌人固定给8分(ENEMY_HIT_SCORE)，没命中给0分

连击倍率 comboMultiplier：
  rawScore >= 6 时连击数+1，否则清零
  倍率 = 连击数为0时1.0，否则COMBO_TABLE[min(连击数-1, 3)] = [1.0, 1.2, 1.5, 2.0]

这一杆实际得分 = round(rawScore × comboMultiplier)
本关累计进度 levelScore += 实际得分
levelScore >= 通关阈值 时立即过关
```

**为什么第二/三关不做环数式的精细命中判定**：三种投掷物用的是完全相同的碰撞检测代码（受保护、这一轮完全没有改动过）——飞行物穿过`z = TARGET_POS.z`这个平面时的精确坐标，逐帧检测，跟以前一模一样。第二/三关的敌人(狐狸/狼)刻意让它们的移动路径固定在这同一个z深度上巡逻，这样命中判定直接复用这个坐标去跟敌人当前x/y位置比距离(`tryHitEnemy`)就够了，不需要另起一套碰撞系统。但也因为这样，"命中"在第二/三关就是简单的"打中了/没打中"，没有环数概念——按大小/部位给不同分值目前没有实际依据，容易变成拍脑袋的平衡性调整，所以先用固定分值把整个流程跑起来，后续如果想做"打中头部加成"之类的细分判定，需要在`tryHitEnemy`里加距离分档，不难加，但这一轮没做。

### 生命值机制细节

- 扣命只发生在第二/三关：敌人的x位置进入被保护对象(兔子/小羊)周围`dangerDistance`范围内时触发，触发后敌人立刻消失、`ENEMY_RESPAWN_DELAY`(1.5秒)后从**巡逻路径远离保护对象的那一端**重新出现——第一版实现里敌人是从靠近保护对象的那一端出生的，等于一生成就在危险区里，第一帧就会扣命，这是Playwright实测直接调用关卡切换函数验证时发现的真bug，已经改成从远端出生、往危险区移动，玩家才有完整的一段巡逻时间可以反应。
- 被保护对象在被"够近"触发扣命的同时会播放一个短促的上下弹动动画（衰减的正弦位移，0.4秒），示意"受惊"。
- 命耗尽（3条命扣完）会暂停当前关卡的敌人移动和抓取判定，显示"再来一次"面板，按扳机键重开——只重置生命值/本关进度/敌人位置，**不刷新整个页面**，也**不会**把你送回第一关（除非你当前就在第一关）。

### 教学关（第一关）专属内容

- **幽灵手柄引导**：一个半透明的简化手柄模型（胶囊体+圆环），进VR后自动播放一次"靠近→握持→后摆蓄力→前甩投掷→淡出"的关键帧动画（7个手工给的关键帧，线性插值，没有做骨骼/IK），演示一遍完整的抓取投掷动作。只播一遍，重开关卡不会重播（重播太多次会烦，这是刻意的），只有第一关会播（触发时机是`renderer.xr`的`sessionstart`事件）。
- **首次投掷提示**：进度条下方一行文字"投中靶心得分更高，试试看！"，完成第一次投掷（不管命不命中）后自动隐藏；重开关卡会让提示重新出现（幽灵手柄引导则不会重播，两者处理逻辑刻意不一样）。

### 已知的简化 / 局限（这一轮新增的）

- **打完第三关会循环回第一关**，还没做"全部通关"的专门画面/结算——`onLevelComplete()`里是`(currentLevel + 1) % 3`，如果想要"三关都打完之后有个不一样的结局画面"，需要在这里加判断，这次没做。
- **第二/三关的命中判定是固定分值(8分)，不区分打中敌人的哪个部位**，跟第一关的环数判定精细度不一样——原因见上面"得分公式"里的说明。
- **关卡数值参数完全没有做过真机平衡性测试**，纯粹是"逻辑跑得通"的估计值，具体见上面的参数表。
- **敌人的移动只有水平巡逻+速度变化，没有任何视觉上的"抬头/警觉"之类的细节动画**——保持了"程序化简单几何体，不追求写实"的一贯风格。

## 第二关真机体验反馈修复（2026-07-10 第六轮）

这一轮是针对"第二关·番茄护兔"的一批具体真机反馈，逐条修的都是**根因**，不是表面调参。改动过程中每完成一步都重新grep核对过7个受保护物理常量（`GRAB_THRESHOLD`/`RELEASE_THRESHOLD`/`PRE_GRAB_THRESHOLD`/`HISTORY_LEN`/`GRAVITY`/`GRAB_BUTTON_INDEX`/`MAX_FLIGHT_TIME`）和碰撞检测代码块（`crossedPlane`那一段），没有被这一轮任何改动碰到；第一关全程用Playwright模拟投掷验证过，体验没有受影响。

**1. 进度条（以及其他所有相机挂载HUD）真机完全不可见——找到了确凿根因**：这是这一轮影响最大的一个bug，而且比"进度条"本身严重得多。按你的要求，没有直接改进度条的代码，而是先用Playwright给一份不提交的调试副本加了运行时探针（`scene.traverse()`遍历检查`camera`是不是真的在场景图里），实测发现：**`camera`这个对象从创建到现在，从来没有被`scene.add(camera)`过**。Three.js的`WebGLRenderer.render(scene, camera)`内部的`projectObject()`只从`scene`开始递归遍历渲染列表（读了`three@0.164.0`的源码逐行确认），`camera`不在这个遍历范围内的话，挂在它下面的所有子物体——生命值心形、进度条、首次提示文字、道具选择面板、退出VR面板、过关/失败全屏遮罩、结算面板，一共8个`camera.add(...)`出来的HUD——**全部不会被渲染，不分桌面还是真机，从这些功能第一次做出来就是这样**。之前几轮的Playwright验证显然只查了"对象存在于`camera.children`里"，没有做过真正的像素级可见性验证，才会一直没发现。用调试副本加`scene.add(camera)`一行测试后，之前完全空白的画面瞬间同时出现了心形/进度条/提示文字/选择面板/退出按钮，一次性实锤。**修复只有一行**：在`camera`创建后加`scene.add(camera)`。加完之后又用真实（非调试）文件跑了一遍完整流程截图确认，同时验证了这一行对WebXR的相机同步没有副作用——`WebXRManager.updateCamera()`内部按`camera.parent`是否为`null`选择不同的矩阵计算分支，`parent`从`null`变成`scene`后，代码走`cameraXR.matrixWorld = scene.matrixWorld * camera.matrix`这条分支，而`scene.matrixWorld`恒为单位矩阵（场景根节点从来没有被移动过），数学上跟原来`parent===null`时的`cameraXR.matrixWorld = camera.matrix`完全等价，不影响真机头显姿态同步。
   - 顺带修了一个次生的清晰度问题：进度条原来的`y=0.34`在`FOV=70°`/`z=-0.55`的视锥里已经非常接近视野上边界（视锥半高约0.385），标题文字顶部会被裁切，桌面截图能看到"第1关"三个字上半截被切掉。改成`y=0.29`（提示文字相应从`y=0.255`挪到`y=0.175`，避免跟进度条重叠），留出安全余量，降低真机FOV跟桌面预览不完全一致时被裁得更多的风险。

**2. 狐狸/兔子模型认不出来——狐狸之前完全没有耳朵**：走查`buildFox()`发现之前的模型只有"身体(球)+头(锥)+尾巴(锥)"，**没有耳朵**——一个光溜溜的橙色球谁都认不出是狐狸，这是"太抽象"反馈的主因。这次加了：狐狸两只三角尖耳朵（`ConeGeometry`）、一块白色肚皮色块（贴在身体下方，橙+白撞色跟纯白兔子拉开辨识度）、尾巴从单薄的细长圆锥换成"粗圆锥+球形尾尖"堆叠出蓬松感；兔子加了一颗小尾巴（之前完全没有），耳朵加长（0.09→0.12），身体颜色调得更接近纯白（`0xf2f2ee`→`0xf5f5f0`）跟狐狸的橙色系拉开对比。改完用Playwright读取`foxGroup`/`rabbitGroup`的实际mesh构成核对过：狐狸现在是7个mesh（身体/肚皮/头/2只耳朵/尾根/尾尖），兔子是4个mesh（身体/2只耳朵/尾巴），颜色分别是橙色系(`#d9701f`/`#faf3e6`/`#f5ead4`)和纯白系(`#f5f5f0`)，结构上跟需求描述一致。**如实说明一点**：森林场景本身是刻意调暗的氛围光照，实际光照下这两个模型的颜色饱和度看起来会比色值本身暗一些，这是场景整体氛围的问题，不是模型颜色选错了——下一轮"森林氛围色调调整"会专门处理这个，这一轮先保证形状轮廓能被认出来。

**3. 狐狸移动路径太短太直——改成途经点巡逻+停留**：原来是两点间(`pathMinX`~`pathMaxX`)匀速直线来回，路径只有2.4m、全程不停，很容易一晃眼整个来回就错过了瞄准窗口。改成`updateWaypointMovement()`：狐狸按固定顺序在5个途经点`[1.3, 0.6, -0.1, -0.8, -1.3]`之间移动，每到一个途经点停留`pauseDuration=1.3`秒（期间有一个轻微上下浮动的待机动画，`呵呵笑`的感觉），停留结束后继续朝下一个点走，循环往复。路径总长拉到2.6m，且因为多次停留，从生成到抵达危险区的总耗时明显变长，同时停留期间是完整的静止瞄准窗口。**全程严格按预设顺序移动，没有引入任何随机性**——这是你明确要求的，可预测性本身就是这类小游戏的乐趣来源(红白机式设计)，随机移动只会让人觉得"不公平"。用Playwright连续采样了6秒的狐狸x坐标，实测确认了"停在1.3两个采样点→移动到0.6→再停两个采样点→继续移动"这个节奏，不是凭代码走查猜的。第三关(狼)完全没有改动，仍然是原来的直线往返+冲刺逻辑（在`updateEnemy()`里用`config.waypoints`是否存在做了分支，两套逻辑互不干扰）。

**4. 难度缓冲——放宽命中判定半径+放大视觉体积**：`tryHitEnemy()`原来固定用全局`ENEMY_HIT_RADIUS=0.18m`，现在改成优先读`enemyConfig.hitRadius`（没配置就退回默认值），第二关配置成`0.24m`（放宽约33%），第三关没配置，沿用原来的0.18m不变。另外`foxGroup`整体视觉缩放到1.2倍（`foxGroup.scale.setScalar(1.2)`），只影响看起来的大小，不影响上面单独控制的命中半径——不追求还原真实狐狸体型比例，优先保证移动靶容易被看清/瞄准，这是你给的两个选项（放大视觉/放宽判定）里都采纳了。

**5. 番茄脱靶没有落点反馈**：之前`resolveThrow()`只在真正命中狐狸时才调`spawnTomatoSplat()`，脱靶（番茄确实穿过了靶平面，但没打中狐狸）完全没有任何视觉反馈，玩家不知道自己偏了多少、偏向哪边。现在在`hitSomething`为`false`但`hitPoint`存在（番茄确实飞到了靶平面，只是没打中）的分支里，复用同一个`spawnTomatoSplat()`，落在番茄实际穿过靶平面的坐标上——作用类似第一关的命中标记点，给一个"刚才差多少"的诊断锚点。如果番茄中途落地或超时（`hitPoint`为`null`，没有明确坐标），则不生成溅落效果，这种情况没有位置可用。同时给`spawnTomatoSplat()`加了`MAX_SPLAT_EFFECTS=10`的软上限，超过时先淘汰最老的一个再生成新的，避免连击刷屏时网格/贴图堆积（原本命中和脱靶都会生成溅落效果，理论并发数比之前只有"命中才有"高一些，加这个上限保险）。用Playwright模拟了一次脱靶（`splatCount`从0变1）和一次命中（分数正确+8）分别验证过。

**6. 第二关整体耗时过长——修完1、3之后重新评估通关阈值**：按你的要求，没有一上来就调数字，先分析"耗时过长"是不是1、3两个根因的下游症状：进度条之前完全不可见，玩家完全感受不到"还差多少"，主观上会觉得"看不到头"；狐狸之前是一晃而过的直线靶，有效命中率低，进度积累慢。这两个根因修完之后，"耗时感"应该已经明显缓解——但即使命中率因为途经点停留+放宽判定大幅提升，`ENEMY_HIT_SCORE`固定8分/命中、连击顶格2.0倍的算法本身没变，达到原来的500分阈值理论上仍然需要约33次命中（全程不脱靶的理想情况），对"三关里的中间一关"来说绝对数量偏多，横向对比第一关(300)/第三关(800)的曲线也偏陡。所以在1、3修完之后，把`LEVEL_THRESHOLDS[1]`从500下调到400（约25次命中），保持"比第一关难、比第三关简单"的相对关系，但把绝对耗时拉回合理区间。这是本轮**唯一**直接调整的数值，而且是在两个根因修复之后才做的，不是替代性的"头痛医头"。

## 真机反馈第七轮修复（2026-07-11）

这一轮是第六轮上线后紧接着的又一批Pico真机实测反馈，三个问题，逐条排查+修根因。同样全程核对过7个受保护物理常量和`crossedPlane`碰撞检测代码块没有被动过（改动前后`grep -c`计数一致，仍是26处引用），第一关用Playwright完整验证过没有受影响。

**1. 所有UI浮层离眼睛太近，手柄射线瞄不准**：走查发现所有相机挂载面板（心形血量/进度条/提示文字/道具选择/退出VR）统一用的是`z=-0.55`，不到0.6米。这些面板是`camera`的子物体，跟头一起转，但手柄是独立6DOF追踪的，物理上跟头有一段真实距离——面板离头越近，手柄射线要精确对上它所需要的角度窗口就越窄，稍微一偏手就完全指不到，这是真实的几何问题，不是代码bug。按你的建议（只读展示类不用改，需要手柄交互的必须改）：
- **心形血量/进度条/提示文字**：只读展示，不需要射线瞄准，维持`z=-0.55`不变
- **道具选择面板**(`SELECT_PANEL_OFFSET`)和**退出VR面板**(`shared/xrExitButton.js`的`OFFSET`)：挪到`z=-1.3`（约1.3米，落在WebXR UI常见的1-2米经验区间），`WIDTH`/`HEIGHT`/`OFFSET`的x/y都按同样的倍数(1.3/0.55≈2.364)放大，这样从摄像机看过去的视觉大小/屏幕位置基本不变，只是换取手柄更宽松的瞄准角度容差。退出按钮是`shared/`下的公共组件，demo0/demo1也在用同一份文件，这一处改动对三个demo都生效——这是同一个根因，不是只有demo-dart才有这个问题，一次性在源头修比每个demo各改一遍更合理。用Playwright读取了两块面板的世界坐标，实测离摄像机的直线距离分别是1.600m（道具选择）和1.643m（退出按钮），确认在目标区间内。**这一项没有真机条件验证，1.3米这个距离是否合适，需要下一轮真机测试确认**——如果还是太远/太近，直接改这两处的距离常量即可，不涉及任何受保护逻辑。

**2. 狐狸路径真机上仍然是直线——重新核实后发现是上一轮验证方法本身的疏漏，不是部署问题**：这条反馈很重要，按你的要求没有直接假定"已经修过了就不用管"，而是重新查了一遍：
   - 先排除了部署/缓存问题——`git log`确认最新commit已经推送，`curl`直接请求生产环境的URL（带时间戳参数强制绕过缓存）核对了`waypoints:`和`LEVEL_THRESHOLDS`的实际内容，**确认部署的就是最新代码，不是缓存滞后**。
   - 排除部署问题后，回头重新看上一轮的5个"途经点"`[1.3, 0.6, -0.1, -0.8, -1.3]`——**这5个数值本身只是x坐标，y全程固定在`config.y=0.32`不变**。也就是说上一轮做的其实是"直线上加几个停顿点"，路径的几何形态从头到尾都没离开过一条水平直线，只是加了停顿动作，这跟"路径点巡逻"给人的直觉预期（应该有转折/绕路）完全对不上。真机上看到的正是这个直线本身，不是幻觉，也不是没部署上去。
   - **上一轮的验证漏洞**：当时用Playwright连续采样了狐狸的x坐标随时间变化，只验证了"x在按停顿节奏变化"这一个事实，从未检查过y坐标或者路径是否偏离直线——验证的是"有没有用途经点机制"，不是"路径长得像不像途经点巡逻"，这是两件不同的事，这次记下来避免下次再犯同样的疏漏。
   - **这次的修复**：把途经点从单纯的x数值改成`{x, y}`坐标对，5个点分别是`{1.3,0.45} → {0.6,0.15} → {-0.1,0.5} → {-0.8,0.15} → {-1.3,0.35}`，y在0.15~0.5之间大幅摆动，`updateWaypointMovement()`同时对x/y做2D插值(用`Math.hypot`算总距离再按比例分配到两个分量)，这样狐狸走出的是真正的Z字形路径。z坐标继续固定在`TARGET_POS.z`不变——**特意没有加z方向的前后变化**，因为`tryHitEnemy`和扣命判定都只比较x/y，如果贸然让狐狸在z方向偏离碰撞检测所在的那个平面，会导致番茄穿过靶平面判定命中的时机和狐狸视觉上的实际位置脱节（番茄可能视觉上还没碰到狐狸就已经"命中"，或者穿过去了才判定命中），只在x/y平面内绕路不会有这个问题，同时也足够做出明显的曲线/Z字形观感。用Playwright连续采样了8秒的狐狸x/y坐标：y从0.47左右一路摆动到0.13、再到0.50、再到0.16、最后到0.50附近，明确不是一条平线，这次的验证方式补上了y坐标和路径形态的检查，不再只看x。
   - 路径坐标+运动轨迹对照表（方便下次真机测试时直接核对）：

     | 途经点顺序 | 坐标(x, y) | 说明 |
     |---|---|---|
     | 1（出发/安全端） | (1.3, 0.45) | 右侧偏高 |
     | 2 | (0.6, 0.15) | 中右侧下探到低点 |
     | 3 | (-0.1, 0.5) | 中间抬高到全程最高点 |
     | 4 | (-0.8, 0.15) | 中左侧再次下探到低点 |
     | 5（危险区端） | (-1.3, 0.35) | 左侧回到中等高度，同时x已经进入`dangerDistance`范围 |

     连起来看是"高→低→高→低→中"的Z字形/波浪路径，不是直线。每个点之间用`pauseDuration=1.3`秒的停顿分隔（停顿期间有小幅上下浮动的待机动画）。

   - **命中判定体积再放大一倍**：`hitRadius`在上一轮(第六轮)已经从默认的0.18m放宽到0.24m；这一轮是在**0.24m的基础上乘以2**，也就是**0.24 → 0.48m**，不是从最初的0.18m算起（0.18×2=0.36m不是这次的目标值，特意在这里写清楚避免混淆）。0.48m是一个相当宽松的判定半径了，配合Z字形路径+停顿，命中难度应该会大幅下降。

**3. 第三关画面几乎完全漆黑**：这是本轮新反馈，之前的测试记录里没提过。排查过程：
   - 先排除了"灯光对象丢失"的可能——`ambient`/`point`是全局唯一的两个灯光实例，`enterLevel()`每次切关卡只是改它们的`.color`/`.intensity`属性，从来没有`remove`或重新创建过，也没有第二关的清理逻辑会碰到它们，这条排除了。
   - 真正原因是`LEVEL_THEMES[2]`这一档数值本身定得太暗：`ambientIntensity`(0.4)/`pointIntensity`(1.2)比第二关(0.8/2.2)砍了将近一半，`fogNear`/`fogFar`(1.8/7)又比第二关(3/12)收得更紧，两个因素叠加，在桌面显示器上看起来是"更暗更冷"的氛围色，换到真机头显（面板亮度/对比度普遍比桌面显示器低）上就直接看不清了。
   - **验证方法**：没有只做语法检查，而是用Playwright截图+像素级验证——把狼的3D世界坐标通过`camera.project()`投影到屏幕坐标，围绕这个坐标裁出一小块区域截图，分别在"改之前的原始数值"和"改之后"两个版本上截图对比。
   - **第一次尝试**（`ambientIntensity 0.4→0.7`、`pointIntensity 1.2→2.0`、放宽`fogNear/Far`到`3/11`）：截图显示确实没有原来那么黑，但狼的轮廓依然基本融进背景里，只有那个不受光照影响的黄色眼睛小球能看清（`eyeMat`是`MeshBasicMaterial`，不受灯光强度影响，一直很显眼——这也是为什么"完全漆黑"的描述里其实还留了一个亮点，眼睛一直都看得见，看不见的是整个身体轮廓）。像素级验证：统计裁剪区域里"亮度明显高于背景"(阈值20)的像素占比，第一次尝试只从0.5%涨到0.5%，基本没有实质性变化。

## 速度计算逻辑（如果实测手感不对，从这里开始查）

飞镖释放瞬间的初速度由两部分相加：

1. **线性速度**：**最近5帧手柄位置的滑动窗口**（2026-07-10从8帧缩短到5帧，提高响应），取窗口里最早一帧和最新一帧的位置差，除以这两帧之间的时间差，得到"释放前这段时间的平均速度"。没有只用最后一帧的瞬时位移——那样太容易被单帧的追踪抖动带偏方向。
2. **旋转带来的切向速度**：手腕甩动的特征是"朝向变化快、位移距离短"，纯线性位移窗口对这种动作天然不敏感——这是2026-07-10发现的问题（见下方修复说明）。现在同时用同一个窗口算出手柄的角速度（旋转轴+角速度大小），再乘上飞镖抓取点相对手柄的偏移向量（叉乘），算出"因为手柄在转动，飞镖被甩出去的那部分速度"，跟线性速度相加。这是标准的刚体运动学公式：`v_point = v_origin + ω × r`。

面板上会显示"上次初速度"这个数值（单位 m/s），**这是专门留出来给你做手感校准用的**：如果你觉得自己甩得很轻，但数值显示速度很大（或者反过来），说明速度计算逻辑有问题，需要回头调整：
- 滑动窗口的帧数（`HISTORY_LEN`，目前是5帧）——窗口太长会把"蓄力阶段的静止"也算进平均速度里，导致数值偏低；窗口太短又会重新变得对抖动敏感
- 是否需要对计算出来的速度做一次缩放校准（目前完全没有做人为放大/缩小，重力也是标准的9.8 m/s²，都是诚实的物理量，没有为了"看起来好扔"而调参）

## 这次修的三个真机测试反馈问题（2026-07-10）

**2. 加了命中诊断可视化**：之前反馈"分数随位置变化能感受到，但计算可靠性不确定"——光看一个环数数字，没法判断是碰撞检测本身算错了，还是自己对空间位置的主观感觉有偏差。现在：
- 飞镖穿过靶平面的精确碰撞点会生成一个跟命中环数同色的小球标记（金色=10分，红色=8分，以此类推，脱靶是灰色），停留2秒后淡出，最多同时保留3个方便对比

**3. 加了场景内退出VR的方式（这是三个反馈里最重要的一个）**：VRButton默认生成的"EXIT VR"是页面上的DOM按钮，一旦进入沉浸模式用户根本看不到浏览器界面，这个按钮实际不可达——之前demo0/demo1/demo-dart三个页面都有这个问题，进去之后没有能用的退出方式。现在做了一个通用组件 `shared/xrExitButton.js`，三个demo都接入了：视野右下方外侧有一块3D面板，手柄射线指向它、按扳机键（trigger，不是抓握键，避免跟demo-dart的抓取手势冲突）就会调用 `renderer.xr.getSession().end()` 优雅退出。面板位置选在"平时看东西不会碰到、但低头瞟一眼右下就能看到"的地方，没做成正对视野中央那种容易误触的位置。

## 这次修的输入响应灵敏度与交互反馈问题（2026-07-10 第二轮）

**1. 抓取响应不灵敏**（反馈"按键后经常没能响应"）：之前只依赖WebXR原生的`squeezestart`/`squeezeend`事件——这两个事件的实际触发时机由浏览器/运行时自己判定，不透明，也不一定够灵敏，等于"什么时候算按下了"这件事完全不在我们控制范围内。现在改成：如果控制器的WebXR Gamepad暴露了模拟量（`button.value`，0~1连续值，而不是只有布尔的`pressed`），每帧轮询，用一个明显低于"按到底"的阈值（`GRAB_THRESHOLD = 0.6`）触发抓取，响应更快也更可控；原生事件继续保留作为兜底（万一某个输入源真的拿不到模拟量）。同时加了"预抓取"反馈：压力超过`PRE_GRAB_THRESHOLD = 0.3`但还没到抓取阈值时，镖身会有一个呼吸感的发光提示，减少"按了但不知道有没有生效"的挫败感。

**2. 手腕小幅度投掷动作捕捉不到**（反馈"应该更好地去捕捉大动作，避免失败过多"——实际现象是小幅快速动作被漏判，不是要弱化大动作识别）：原来的8帧滑动窗口只算线性位移，对"手腕甩动"这种"朝向变化快、位移距离短"的动作不友好——纯线性位移窗口会把这种动作的力道稀释成一个很小的速度值，达不到有效投掷的门槛。现在做了两处调整：滑动窗口从8帧缩短到5帧（提高响应），并且新增了角速度的贡献——用刚体运动学公式`v_point = v_origin + ω × r`把"手柄转动导致飞镖被甩出去"的那部分切向速度也算进去（详见上方"速度计算逻辑"）。这个公式对大幅度手臂投掷（转动慢，这一项贡献很小）和小幅度手腕甩动（转动快，这一项能补足线性位移算出来的"虚弱"速度）是自然兼顾的，不需要写"看起来是大动作还是小动作"这种启发式判断分支。

**3. 交互反馈层太单薄**（之前只有面板数字变化）：加了音效和视觉两层反馈，都用Web Audio API振荡器/白噪声合成，没有引入外部音频文件依赖：
- 抓取成功：短促降调的"咔"声（方波+快速包络）
- 命中靶面：按环数分级触发不同音调，外环闷（低频sine），内环脆（更高频、triangle音色更亮），10环额外叠加一段带通滤波白噪声模拟的欢呼声
- 视觉上，6环以上会在命中点弹出评级文字（GOOD/GREAT/EXCELLENT），Canvas贴图+简单的弹出缩放动画，1.5秒后连同标记点一起淡出；2/4分和脱靶只有命中标记点，不额外弹字，避免每次投掷都弹一堆文字造成视觉噪音

## 这次修的按键映射与反馈层排查（2026-07-10 第三轮）

**1. 抓取键从抓握键(squeeze/grip)改成扳机(trigger)**：第二轮测试已经确认squeeze的响应机制work得很好（左右手都顺畅），但用户反馈trigger更符合"抓取/发射"类动作的生理直觉——食指扣扳机比中指侧键按压更接近直觉里的"抓东西"动作。第二轮调好的响应机制（阈值滞回0.6/0.45、预抓取发光0.3）原样保留，只是把监听的物理按键源从`buttons[1]`(squeeze)换成`buttons[0]`(trigger)，原生事件也从`squeezestart`/`squeezeend`换成了WebXR规范里trigger对应的标准事件名`selectstart`/`selectend`。**这次没有能力上真机测试，`GRAB_BUTTON_INDEX=0`仍然是标准映射的假设值，没有实测验证过**——已经按你的要求加了`logGamepadButtonChanges`日志（见上方"按键说明"），下次测试时留意控制台输出确认这个假设成不成立。

**2. 退出按钮同步换成抓握键，避开冲突**：`shared/xrExitButton.js`之前特意选trigger是为了避开demo-dart的抓取手势（当时抓取用squeeze），现在抓取改成trigger了，两者会撞在一起，所以退出按钮同步改成了squeeze。确认过这个改动对demo0/demo1没有副作用：demo0没有绑定任何自定义手柄事件；demo1的粒子档位切换绑的是`selectstart`(trigger)，跟squeeze完全不冲突。

**3. 排查音效完全没反应——找到了确凿的根因并修复**：AudioContext的`resume()`只有在真正的"用户手势"调用栈里发起才会生效，这是浏览器的自动播放限制。走查代码发现：之前唯一调用`ensureAudio()`的地方是`grabDart()`，而`grabDart()`同时被两条路径调用——原生的`selectstart`事件（真实手势）和每帧轮询的`updateAnalogGrabInput()`（只是渲染循环里的普通函数调用，不算用户手势）。因为轮询每帧都在跑，几乎总是比原生事件先摸到抓取阈值、抢先调用`grabDart()`；`grabDart()`内部有状态守卫，抢到的那次执行完，原生事件触发时状态已经不是IDLE了，直接return——`playGrabSound()`根本没机会在"真手势"的调用栈里跑。`resume()`在非手势上下文里静默失败、不报错，现象就是"完全没有声音、控制台也没有报错"，跟你的猜测完全一致。**已修复**：把第一次`resume()`尝试挪到"Enter VR"按钮的click事件里——这是唯一100%保证是真实用户手势的时机。同时在`ensureAudio()`里加了`console.log`打印`audioContext.state`的变化，下次测试能直接在控制台确认状态是不是真的变成了`running`。

**4. 排查视觉反馈（GOOD/GREAT/EXCELLENT）没出现——代码走查没找到同等确凿的bug**：跟音效不一样，`resolveThrow()`→`spawnRatingText()`→贴图生成→`scene.add(mesh)`这条调用链逐行看下来没发现明显的"接错线"问题。老实说，这次没有设备可以实测，不能排除有一个还没发现的隐藏bug；但也有一个同样成立的可能性——评级文字只有命中6/8/10环才会弹（2/4分和脱靶只有标记点），如果当时的测试投掷因为抓取/速度问题还不够准，根本没打到6环以上，那视觉反馈"没出现"就是设计上的正常行为，不是bug。**没有实机验证不能瞎猜，所以这次没有重写这块逻辑**，而是按你的要求加了日志：`resolveThrow()`里打印`scoreValue`和`hitPoint`，`spawnRatingText()`里打印是否找到了对应的贴图、mesh有没有成功add到scene。下次测试时如果日志显示函数确实被调用了、mesh也生成了，但VR里还是看不见，那就是真的有渲染层面的bug（比如位置计算飘到视野外之类），到时候可以照着日志的实际数值排查；如果日志显示`scoreValue`每次都小于6，那就说明只是测试投掷没打到高分区，不是代码问题。

## 这次的视觉/玩法升级（2026-07-10 第四轮）

这一轮明确要求"不改动现有的输入判定、速度计算、碰撞检测逻辑，只在这些基础上叠加新内容"——抓取阈值、滑动窗口、角速度公式、逐帧穿越检测这些全部原样未动，改动过程中每完成一步都重新核对过一遍这几个常量/关键代码块没有被意外改到。

**1. 场景氛围改成昏暗森林**：背景/雾从深空蓝黑(`0x05070c`)换成深绿偏黑(`0x070a06`)，雾的近/远距离也从`(3,14)`收紧到`(2,9)`，视线受限的感觉更明显。原来的`GridHelper`地面换成了程序化Canvas噪声贴图（大块深绿苔藓斑块+细颗粒落叶噪点混合），铺在一个20x20的地面Plane上并重复平铺4x4次。灯光配色也从冷色调科幻蓝（`0x445577`环境光+`0x7fd4ff`点光源）换成暗绿环境光+暖黄点光源（模拟树冠缝隙漏下来的光），强度调低配合浓雾维持昏暗基调。周围放了5棵简化树干剪影（圆柱树干+3层圆锥树冠），位置刻意避开靶(z≈-2.5)和飞行轨迹走廊(x≈0)，放在两侧和更远处；几何体segment数很低（6-7段），对帧率影响可以忽略。

**2. 飞行尾焰/火星拖尾**：用固定上限（80个）的`Points`粒子池实现，不是每帧new/delete，而是用`Float32Array`直接原地改写position/color两个`BufferAttribute`。粒子从投掷物尾部（本地+Z偏移0.06处，用`localToWorld`转换成世界坐标）生成，初速度是"反向 + 随机抖动"，飞行速度越快（读当前`dartVelocity.length()`，映射到0~8m/s→0~1的`speedFactor`）单帧生成的粒子数越多（1~3个）。用`AdditiveBlending`叠加混合让重叠的火星有发光感，颜色亮度随粒子年龄线性衰减模拟alpha淡出（配合加法混合，暗=看起来透明，不需要额外管理per-vertex alpha或自定义shader）。颜色按投掷物类型区分（见下一条）。

**3. 三种可选投掷物：飞镖/番茄/斧头**：三个物体在启动时就全部预建好（`buildDart`/`buildTomato`/`buildAxe`），加进场景后只是切换`.visible`，切换类型不会重复创建/销毁几何体。飞镖保持原来的圆锥+圆柱组合一个字没动；番茄是轻微不均匀缩放的球体（`SphereGeometry`+挤压scale）；斧头是圆柱柄+压扁四棱锥刃（`ConeGeometry(radius,height,4)`缩放成扁的楔形）。选择方式是场景内一块小面板（左下方，跟退出按钮的射线检测+hover高亮是同一套实现模式），三个选项各自一个独立的小方块，点亮当前选中项。**这里跟最初"复用trigger点击"的建议不一致**：因为trigger已经是全局的"抓取"键（对任意方向扣扳机都会尝试抓取，没有距离/朝向限制），如果选择面板也绑trigger，指着菜单点选的同时会顺带触发一次抓取，两个操作打架，所以改用了squeeze（详见上方"按键说明"，已经确认对demo0/demo1和退出按钮都没有副作用）。切换只在投掷物处于`IDLE`状态时允许，持有/飞行中切换会被直接忽略，避免半路换物体产生奇怪的状态。

**4. 三种物体的差异化命中效果**：
- **计分逻辑（按要求在这里说明）**：三种投掷物用的是完全相同的"距靶心距离"计分——`RINGS`环数划分、`scoreForDistance()`、逐帧平面穿越检测，一套代码不区分类型。这是刻意选的最简单方案，一方面完全不用碰"不要改动碰撞检测逻辑"这条红线，另一方面三种物体的物理弹道本来就走的是同一个`GRAB_OFFSET`/同一套速度公式（不然会违反"不改动速度计算"的要求），弹道行为本身没有差异，只按大小/形状给不同的计分曲线目前没有实际依据，容易变成"拍脑袋"的平衡性调整。如果以后想让番茄"更容易命中大范围但精度低"、斧头"命中即高分但范围小"这类差异化玩法，需要专门设计新的计分函数，这次没做，先都用统一逻辑跑起来看手感。
- **飞镖**：保持原样，跟三轮改动之前完全一样，没有叠加任何新东西
- **番茄**：命中瞬间生成一个独立的临时"砸扁"效果——前120ms快速压扁（Y轴缩到约1/4，XZ轴撑大到接近2倍），之后维持压扁状态在550ms内淡出；同时生成2~3片红色小平面模拟果肉飞溅，带一点轻力重力往外飞散。这个效果是**独立生成的装饰物**，不是让真正的番茄物体本身变形——真正的番茄命中后跟其他物体一样立刻飞回中性待抓取位置，如果直接在它身上做形变动画，会跟"飞回去"的动作同时发生，观感很怪
- **斧头**：命中瞬间靶面（`targetMesh`+`backboard`）做一个180ms内快速衰减的随机位置抖动，抖动结束后精确复位到`TARGET_POS`（不会有浮点误差累积导致靶子慢慢跑偏的问题）
- **音效**：飞镖保持原来的6档环数分级音调；番茄是一个独立的低通滤波音调（`BiquadFilter` lowpass在400Hz）+短促噪声，听感发闷，不分环数；斧头是极低频音调（110→70Hz）+短促噪声模拟"咔"的钝重感，也不分环数。三种物体的命中音效不管命不命中靶面（`hitPoint`是否为null）都会响，保持"每次投掷都有声音反馈"的行为一致

## 已知的简化 / 局限

- **扳机键在xr-standard手柄映射里假设是按钮索引0**（`GRAB_BUTTON_INDEX = 0`，索引1是抓握键squeeze，现在给退出按钮用）——这是WebXR标准手柄profile的通用约定，2026-07-10这次改动加了`logGamepadButtonChanges`实测日志（见上方"按键说明"），下次测试时留意控制台输出，如果"预抓取发光提示一直不亮"或者索引对不上，日志里能直接看到实际按下的是哪个`buttons[i]`
- 碰撞检测是逐帧检查飞镖是否穿过靶所在的平面，不是连续碰撞检测——如果初速度极快导致单帧内穿越距离过大，理论上可能出现"隧穿"漏判的情况，先不处理，除非实测发现这是个真问题
- 没有接入 cannon.js 之类的完整物理引擎，重力+速度是手写的逐帧积分，够用但不是精确仿真
- 两只手柄不能同时抓住飞镖，飞镖同一时刻只能被一只手持有
